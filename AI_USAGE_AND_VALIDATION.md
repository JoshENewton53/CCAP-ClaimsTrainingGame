## Overview

This document describes every place artificial intelligence and machine learning are used in the CCAP Claims Training Game, what each component does, how it was built, and how its outputs are validated. The application uses two AI components: a locally trained XGBoost model for claim classification and confidence scoring, and a locally hosted **Google Flan-T5** large language model that generates natural language feedback explaining each claim decision to the user.

---

## 1. AI Models

### 1.1 Google Flan-T5 — Natural Language Feedback Model
**Model**: `google/flan-t5-base` (HuggingFace)
**Size**: ~250 MB (downloaded and cached locally on first run)
**Library**: HuggingFace `transformers` + `sentencepiece`
**Loaded by**: `load_flan_t5()` in `ai_service.py`

**What it does:**
Flan-T5 is a sequence-to-sequence large language model developed by Google and released publicly on HuggingFace. It was pre-trained on a massive corpus of text and then fine-tuned on hundreds of instruction-following tasks, giving it the ability to read a prompt and generate coherent, contextually appropriate natural language responses.

In this application, Flan-T5 is used to generate the personalized feedback explanation a user sees after submitting their answer. Rather than returning a hardcoded string, the model receives a structured prompt describing the specific claim, the user's answer, the correct answer, and the key finding from the business rule validation layer. It then generates a 2–3 sentence explanation in plain English.

**Example prompt sent to the model:**
```
You are an insurance claims training assistant.
A trainee reviewed a dental insurance claim (procedure: D1110, diagnosis: K04.5, amount: $320.00)
and answered 'valid'. The correct answer is 'invalid'.
Key finding: Procedure D1110 requires diagnosis codes K02.9, K05.10, K04.7. The claim shows K04.5, which is not valid.
In 2-3 sentences, explain what they missed and why the correct answer is invalid.
Be specific to this claim type and educational.
```

**Example output generated by Flan-T5:**
> "The procedure code D1110 represents a routine dental cleaning, which is a preventive service. Preventive procedures must be paired with preventive diagnosis codes — K04.5 refers to a pulp condition, which is not appropriate for a cleaning claim. Always verify procedure-diagnosis pairings using the Code Reference before approving a dental claim."

**Where it is used in the codebase:**
`ai_service.py` → `generate_flan_feedback()`, called from `generate_feedback()`, which is called from `app.py` → `/api/scenario/submit`

**Fallback behavior:**
If Flan-T5 fails to load or throws during inference, `generate_feedback()` automatically falls back to the rule-based `generate_explanation()` function so the application continues working.

### 1.2 XGBoost Claim Classifier
**File**: `ai_models/models/claim_classifier.pkl`  
**Size**: ~88 MB  
**Training script**: `ai_models/train_classifier.py`  
**Dataset**: `datasets/claims_training_data.csv` (1,000+ labeled insurance claim records)

**What it does:**  
Classifies any insurance claim scenario as one of three outcomes: `valid`, `invalid`, or `insufficient`. It is the core decision-making model of the application.

**Input features the model was trained on:**
- Claim amount (numeric)
- Patient age (numeric)
- Claim type (encoded: medical / dental / life)
- Procedure code (label-encoded)
- Diagnosis code (label-encoded)
- Sentence embedding of a natural language description of the claim (generated by Sentence Transformers during training)

**Output:**
- Predicted label (`valid`, `invalid`, or `insufficient`)
- Confidence score (0.0 – 1.0)
- Probability distribution across all three classes
- AI reasoning string summarizing the decision

**Where it is used in the codebase:**
`ai_service.py` → `classify_claim()`, called from `app.py` → `/api/scenario/generate` and `/api/scenario/submit`

## 2. How the Models Work — Code Walkthrough

### 2.1 How Flan-T5 Generates Feedback (`generate_flan_feedback()` in `ai_service.py`)

Every time a user submits an answer, `generate_feedback()` is called. It first runs `generate_explanation()` to extract the key finding from the scenario using business rules (e.g. "procedure D1110 does not match diagnosis K04.5"). That finding is then passed to `generate_flan_feedback()` as context.

**Step 1 — Build the prompt**
```python
prompt = (
    f"You are an insurance claims training assistant. "
    f"A trainee reviewed a {claim_type} insurance claim "
    f"(procedure: {procedure}, diagnosis: {diagnosis}, amount: ${amount:.2f}) "
    f"and answered '{user_answer}'. The correct answer is '{correct_answer}'. "
    f"Key finding: {rule_explanation} "
    f"In 2-3 sentences, explain {'why they were correct' if is_correct else 'what they missed and why the correct answer is ' + correct_answer}. "
    f"Be specific to this claim type and educational."
)
```
The prompt is constructed dynamically from the actual scenario data so every explanation is unique to that specific claim.

**Step 2 — Tokenize and run inference**
```python
inputs = _flan_tokenizer(prompt, return_tensors='pt', max_length=300, truncation=True)
outputs = _flan_model.generate(
    **inputs,
    max_new_tokens=120,
    num_beams=4,
    temperature=0.7,
    repetition_penalty=1.3
)
return _flan_tokenizer.decode(outputs[0], skip_special_tokens=True)
```
Beam search (`num_beams=4`) explores multiple candidate sequences and selects the best one, producing more coherent output than greedy decoding. The `repetition_penalty` prevents the model from repeating phrases.

**Step 3 — Fallback**
If the model is not loaded or throws an exception, `generate_flan_feedback()` returns `None` and `generate_feedback()` uses the rule-based explanation string instead.

### 2.2 Training the XGBoost Model (`ai_models/train_classifier.py`)

The XGBoost classifier is a **gradient boosting** model, meaning it builds a sequence of decision trees where each tree learns to correct the mistakes of the ones before it. The final prediction is a weighted vote across all 200 trees. This approach is well suited to tabular data like insurance claims where the relationships between fields (procedure code, diagnosis code, claim amount, age) determine validity.

The training pipeline works in four steps:

**Step 1 — Load and split the data**
```python
train, val, test = load_data()  # splits from claims_training_data.csv
```
The dataset is pre-split into training, validation, and test sets. The model only ever sees the test set once at the end to get an unbiased accuracy reading.

**Step 2 — Encode categorical features**
```python
encoders['claim_type'] = LabelEncoder()
train['claim_type_enc'] = encoders['claim_type'].fit_transform(train['claim_type'])
```
XGBoost requires numeric inputs. scikit-learn's `LabelEncoder` converts text categories like `"medical"`, `"dental"`, `"life"` and all procedure/diagnosis codes into integers. The fitted encoders are saved alongside the model so the same mapping is used at runtime.

**Step 3 — Generate sentence embeddings**
```python
model = SentenceTransformer('all-MiniLM-L6-v2')
train_embeddings = model.encode(train_texts, show_progress_bar=True)
```
Each claim is converted into a natural language string (e.g. `"medical claim for 99213 diagnosis I10 amount $450 age 52"`) and passed through a pre-trained Sentence Transformer neural network. This produces a 384-dimensional vector that captures semantic meaning. These embeddings are concatenated with the structured numeric features to form a hybrid feature vector — giving the model both hard numbers and language context.

**Step 4 — Train the XGBoost classifier**
```python
model = XGBClassifier(
    n_estimators=200,    # 200 trees in the ensemble
    max_depth=4,         # each tree can be at most 4 levels deep
    learning_rate=0.05,  # small steps to avoid overfitting
    subsample=0.8,       # each tree sees 80% of the training data
    colsample_bytree=0.8
)
model.fit(X_train, y_train, eval_set=[(X_val, y_val)])
```
Regularization parameters (`reg_alpha`, `reg_lambda`, `min_child_weight`) prevent the model from memorizing the training data. The validation set is monitored during training. The final model, encoders, and embedding model are all saved together into `claim_classifier.pkl`.

### 2.3 Loading the Models at Runtime (`ai_service.py`)

When the Flask backend starts, `ai_service.py` loads the saved model file into memory once:

```python
with open(CLASSIFIER_PATH, 'rb') as f:
    data = pickle.load(f)
    _classifier = data['model']                    # the XGBoost model
    _encoders = data['encoders']                   # the LabelEncoders
    _embedding_model = data.get('embedding_model') # Sentence Transformer
```

All three components are stored as module-level globals so they are loaded once on startup and reused for every request without reloading from disk.

### 2.4 Running a Prediction (`classify_claim()` in `ai_service.py`)

Every time a scenario is generated or a user submits an answer, `classify_claim()` is called. Here is what happens inside it:

**Step 1 — Business rule checks run first**
Four deterministic validation functions run before the model is ever called. If any fire, the answer is returned immediately with 100% confidence and the model is skipped:
```python
if not validate_age_matching(scenario_data):  # DOB vs claim age
    return {"prediction": "invalid", "confidence": 1.0, ...}
if not validate_injury_date(scenario_data):   # injury before policy start
    return {"prediction": "invalid", "confidence": 1.0, ...}
if not validate_code_matching(scenario_data): # procedure/diagnosis pairing
    return {"prediction": "invalid", "confidence": 1.0, ...}
if has_missing_critical_documents(...):       # required docs absent
    return {"prediction": "insufficient", "confidence": 1.0, ...}
```

**Step 2 — Safety check for unseen labels**
If a procedure or diagnosis code was not in the training data, the `LabelEncoder` would crash. The code checks for this before encoding:
```python
known_procedures = set(_encoders['procedure_code'].classes_)
if scenario_data['procedure_code'] not in known_procedures:
    return classify_with_rules(scenario_data)  # fall back to heuristics
```

**Step 3 — Build the feature vector**
The same hybrid feature construction used during training is reproduced at inference time:
```python
structured = np.array([[
    scenario_data['claim_amount'],
    scenario_data['patient_age'],
    claim_type_enc,
    procedure_code_enc,
    diagnosis_code_enc
]])
embedding = _embedding_model.encode([text])      # 384-dim vector
features = np.hstack([structured, embedding])    # combined input to model
```

**Step 4 — Get prediction and probabilities**
```python
prediction = _classifier.predict(features)[0]
probabilities = _classifier.predict_proba(features)[0]
predicted_label = _encoders['label'].inverse_transform([prediction])[0]
```
`predict_proba()` returns a probability for each of the three classes (e.g. `{valid: 0.12, invalid: 0.81, insufficient: 0.07}`). The highest probability becomes the prediction and is shown to the user as the confidence score after they submit their answer.

## 3. AI-Powered Features

### 3.1 Claim Classification and Correct Answer Determination
**Function**: `classify_claim()` in `ai_service.py`  
**API endpoint**: Called internally during `/api/scenario/generate`

When a new scenario is generated, the XGBoost classifier runs on the full scenario data (including client profile and document status) to determine the ground-truth correct answer. This answer is stored in the database and used to score the user's submission.

The classification pipeline:
1. Checks for label-encoder compatibility (unseen codes fall back to rule-based classification)
2. Encodes categorical features using the saved `LabelEncoder` objects from training
3. Builds a hybrid feature vector: structured numeric features + sentence embedding
4. Runs `predict()` and `predict_proba()` on the XGBoost model
5. Returns prediction, confidence, probability distribution, and reasoning

Business rule overrides are applied before the model runs for cases where the answer is deterministic (e.g., confirmed age mismatch, confirmed pre-existing condition, confirmed code mismatch, confirmed missing critical documents). This ensures the model is not asked to classify cases where the answer is already mathematically certain.

---

### 3.2 AI Confidence Scoring on Answer Submission
**Function**: `classify_claim()` in `ai_service.py`  
**API endpoint**: `/api/scenario/submit`

After a user submits their answer, the classifier runs again on the same scenario and its output is returned to the frontend alongside the feedback. The user sees:
- What the AI predicted
- The AI's confidence percentage
- The probability split across all three outcomes
- Whether the AI agreed with the user's answer
- A natural language reasoning string

This gives users a second perspective on their decision and helps them understand borderline cases.

---

### 3.3 AI-Generated Feedback and Explanations
**Functions**: `generate_feedback()`, `generate_flan_feedback()`, `generate_explanation()` in `ai_service.py`
**API endpoint**: `/api/scenario/submit`

After a user submits their answer, the application generates a personalized explanation using a two-layer approach:

**Layer 1 — Business rule extraction (`generate_explanation()`)**
The rule layer inspects the actual scenario data in priority order to identify the key finding:
1. **Age mismatch** — calculates the patient's true age from the DOB in the client profile and compares it to the age listed on the claim.
2. **Pre-existing condition** — compares the injury/service date to the policy start date.
3. **Procedure-diagnosis code mismatch** — looks up the procedure code in `reference_data/code_mappings.json` and checks whether the diagnosis code is in the valid set.
4. **Missing critical documents** — checks the document status against the required critical documents for the claim type.
5. **Valid claim** — confirms all checks passed.

**Layer 2 — Flan-T5 natural language generation (`generate_flan_feedback()`)**
The key finding from Layer 1 is passed to the Flan-T5 language model as part of a structured prompt. The model generates a 2–3 sentence explanation in plain English that is specific to the claim type, the user's answer, and the actual codes and values in the scenario. This is the explanation the user sees on screen.

### 3.4 Contextual Hint System
**Function**: `generate_ai_hint()` in `ai_service.py`  
**API endpoint**: `/api/ai/hint`

When a user clicks "Get AI Hint," the backend retrieves the full stored scenario (including data hidden from the user such as the client profile's true DOB and the injury date) and runs four detection checks:

| Check | What it inspects | Example hint produced |
|---|---|---|
| Age mismatch | Claim age vs. calculated age from DOB | "The claim lists the patient as 45 years old. Compare that carefully with the date of birth in the Client Profile." |
| Timeline issue | Injury date vs. policy start date | "Look at the Date of Injury/Service (2021-03-12) and compare it to when the policy became active." |
| Code mismatch | Procedure code vs. valid diagnosis codes in reference data | "Procedure 99213 is on the claim. Use the Code Reference to check which diagnosis codes are valid for that procedure." |
| Missing documents | Missing list vs. critical document requirements | "Check the Document Status section — some items listed as missing may be required before this claim can move forward." |

Hints are progressive: the first request returns a vague directional hint pointing at the area of the problem without naming it. Subsequent requests return a more specific hint with actual values from the scenario. The answer itself is never stated.

**Future Development:**
The hint system is architected in a way that makes it straightforward to upgrade into a fully generative AI component. Because the backend already retrieves the complete scenario object — including all data hidden from the user — a future version could construct a structured prompt containing everything the user sees alongside the hidden answer context, then pass it to a large language model (such as Claude via AWS Bedrock, or OpenAI's GPT-4) with an instruction not to reveal the answer directly. The LLM would then generate a free-form, conversational hint tailored to that specific scenario rather than selecting from pre-written templates. The API endpoint, the frontend button, and the progressive attempt logic would all remain unchanged — only the internals of `generate_ai_hint()` would need to be swapped out.

### 3.7 Death Certificate Scenario Generation
**File**: `backend/death_certificate_service.py`  
**API endpoint**: `/api/death-certificate/generate`

For life insurance claims, the application generates a realistic death certificate PDF with controllable errors. The service selects errors based on difficulty level:
- Easy: 0–1 errors
- Medium: 0–2 errors
- Hard: 0–3 errors

Possible error types include name mismatch, date inconsistency, missing physician signature, invalid cause-of-death code, jurisdiction mismatch, age mismatch, state mismatch, and expired certification. The user reviews the certificate and identifies which errors are present. Their findings are scored against the actual injected errors.

### 3.8 Itemized Bill Generation with Error Injection
**File**: `backend/itemized_bill_service.py`

For medical and dental claims, the application generates a realistic itemized bill. On medium and hard difficulty, the service probabilistically injects billing errors:
- Wrong CPT/CDT code on a line item
- Math error where the line item total does not match unit cost × quantity
- Duplicate charge for the same procedure

The user is expected to notice these discrepancies as part of their claim review.

## 4. Validation Strategy

### 4.1 Business Rule Validation Layer
Before the ML model runs, a set of deterministic business rules validate the scenario. These rules always override the model when triggered:

| Rule | Implementation |
|---|---|
| Age matching | `validate_age_matching()` — calculates age from DOB, flags >1 year difference |
| Injury date | `validate_injury_date()` — flags injury date before policy start date |
| Code matching | `validate_code_matching()` — checks `code_mappings.json` for valid procedure-diagnosis pairs |
| Missing critical docs | `has_missing_critical_documents()` — checks against per-claim-type required document lists |

This layered approach means the model is only used for probabilistic judgment on genuinely ambiguous cases. Clear-cut violations are caught by rules with 100% confidence.

### 4.2 Label Encoder Safety Check
Before passing data to the XGBoost model, the code checks whether the claim type, procedure code, and diagnosis code were all present in the training data. If any value is unseen, the model is skipped and the rule-based fallback (`classify_with_rules()`) is used instead. This prevents `sklearn` `LabelEncoder` from throwing on unknown categories.

```python
# From ai_service.py - classify_claim()
known_claim_types = set(_encoders['claim_type'].classes_)
known_procedures = set(_encoders['procedure_code'].classes_)
known_diagnoses = set(_encoders['diagnosis_code'].classes_)

if (scenario_data['claim_type'] not in known_claim_types or ...):
    return classify_with_rules(scenario_data)
```

### 4.3 Graceful Degradation
Every AI feature has a fallback so the application remains fully functional if the model is unavailable:
- If the classifier model fails to load → `classify_with_rules()` uses difficulty-based heuristics to determine the correct answer
- If the classifier throws at runtime → same rule-based fallback is invoked automatically
- If hint generation throws → a generic review prompt is returned to the user
- If itemized bill generation fails → the UI displays a warning and continues without the bill

### 4.4 Model Training Validation
The XGBoost classifier was evaluated on a held-out test set during training. The training script (`ai_models/train_classifier.py`) reports accuracy, precision, recall, and F1 score per class. Reported test accuracy is approximately 85%.

## 8. Honest Assessment

The following is an honest account of what the AI does and does not do in this project:

**What the AI genuinely does:**
- **Flan-T5** is a real large language model (encoder-decoder transformer architecture) that reads a dynamically constructed prompt and generates free-form natural language output. The feedback text a user sees after submitting an answer is generated by the model at inference time — it is not a template or a lookup table.
- The XGBoost classifier was trained on real labeled data and produces probabilistic outputs with confidence scores. It is a real trained model, not a lookup table.
- The hint system uses the same validation logic that powers `classify_claim()` to inspect the actual scenario data and produce hints that are specific to each individual scenario's real content — specific ages, dates, and codes from that scenario.
- The confidence scoring shown after submission reflects the model's genuine probability distribution across all three outcome classes.

**Where rule-based logic is used alongside the models:**
- The correct answer for deterministic cases (confirmed age mismatch, confirmed code mismatch, etc.) is determined by business rules before the XGBoost model runs. The model handles genuinely ambiguous cases.
- Flan-T5 receives the key finding from the rule layer as part of its prompt. This grounds the model's output in factually correct information about the scenario rather than relying on the model to independently reason about insurance rules.
- The hint text is constructed from templates with real scenario values inserted, rather than free-form generated text.
- A rule-based fallback (`classify_with_rules()`) is available for any scenario where XGBoost cannot run, and a rule-based explanation fallback is available if Flan-T5 cannot run.
